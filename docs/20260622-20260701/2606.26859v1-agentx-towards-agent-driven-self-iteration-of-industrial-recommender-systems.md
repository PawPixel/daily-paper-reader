## 摘要
推荐算法的迭代正从一种依赖工程师的手工过程转向工业化的研究循环，但这一转变仍受到结构性执行瓶颈的阻碍：从想法到上线的周期仍然依赖人类工程师提出假设、修改生产代码、启动A/B实验并归因线上结果。因此，创新规模随人头数线性增长，而非随证据、算力和积累的实验知识复合增长。我们提出AgentX，一个在生产环境中部署的多智能体系统，从根本上重构了这一生产函数。AgentX作为一个自我进化的开发引擎运作：它以任何手动工作流无法维持的规模和速度自主生成、实现、评估推荐实验并从中学习。

该系统在一个闭环中编排四个紧密耦合的阶段。头脑风暴智能体综合来自历史实验、系统架构、数据分析和外部研究的证据，形成排序后的可执行提案。开发智能体通过基于仓库的生成和多维可靠性验证，将每个提案转化为可投入生产的代码。评估智能体通过带护栏否决的A/B判断进行安全的在线发布，将成功和失败都转化为结构化的知识资产。随后，一个驾驭进化层（SGPO）将执行轨迹提炼为语义梯度更新，持续提升智能体自身——使系统不仅自动化，而且能自我改进。

## Abstract
Recommendation algorithm iteration is moving from an artisanal, engineer-bound process toward an industrialized research loop, but this transition remains blocked by a structural execution bottleneck: the idea-to-launch cycle still depends on human engineers to generate hypotheses, modify production code, launch A/B experiments, and attribute online results. Innovation therefore scales linearly with headcount rather than compounding with evidence, compute, and accumulated experimental knowledge. We present AgentX, a production-deployed multi-agent system that fundamentally restructures this production function. AgentX operates as a self-evolving development engine: it autonomously generates, implements, evaluates, and learns from recommendation experiments at a scale and pace that no manual workflow can sustain.   The system orchestrates four tightly coupled stages in a closed loop. A Brainstorm Agent synthesizes evidence from historical experiments, system architecture, data analysis, and external research into ranked, executable proposals. A Developing Agent translates each proposal into production-ready code through repository-grounded generation and multi-dimensional reliability verification. An Evaluation Agent conducts safe online rollout with guardrail-vetoed A/B judgment, converting both successes and failures into structured knowledge assets. A Harness Evolution layer (SGPO) then distills execution trajectories into semantic-gradient updates that continuously sharpen the agents themselves -- making the system not merely automated, but self-improving.

## 二、论文提出的方法论

### 2.1 核心思想
AgentX是一个**自进化的开发引擎**，由多个角色专精的智能体组成闭环：
- **头脑风暴智能体**：将模糊意图转化为证据支持的、可执行的实验提案。
- **开发智能体**：将提案转化为生产级代码，支持在线策略和离线模型两种轨道。
- **评估智能体**：安全发布、A/B判断，将成功与失败都转化为结构化知识。
- **驾驭进化层（SGPO）**：通过执行轨迹学习，更新智能体的提示词、规则和工具使用规范，实现系统自我改进。

### 2.2 关键技术细节

#### 头脑风暴智能体
- **工作流程**：
    1. **问题边界设定**：将用户意图标准化为明确的输入边界（目标、业务范围、禁止变更、已知约束等）。
    2. **有界提案探索**：分批生成候选想法，按成熟度分为三类：可实施、需前置探测、战略储备。每轮后记录已拒绝的方向形成避免集，下一轮搜索剩余空间。
    3. **证据加权生成**：候选分数由多个维度加权组合：目标对齐度、业务语义有效性、实现可行性、交接完整性、证据强度（来自四个知识源）减去风险惩罚。
    4. **四个知识源**：
        - **实验知识库**：历史发布审查、业务定义、实验教训。
        - **系统知识库**：模型架构、特征定义、DSL行为、管道边界、源代码范围。
        - **数据分析**：历史分析报告、SQL结果、离线统计。
        - **模型研究**：论文结构化的知识（声明、组件、关系），并标注证据强度和工业兼容性。
    5. **验证和交接**：经过人类审查和自动检查（参数冲突、知识库一致性、代码范围、边界检查）后，通过验证的想法形成正式实验记录交给开发智能体。

#### 开发智能体（在线策略轨道）
- **挑战**：生产代码的可靠性问题——属性幻觉、DSL误用、注册模式缺失、特征开关失当等。
- **解决方案**：
    1. **基于仓库的生成**：使用项目特定知识库（包含变更模式、注册惯例、特征开关规则和已接受补丁示例）和案例工具箱（模式查询工具、DSL检查器、C++语法检查器、轻量级静态Linter）。
    2. **面向验证的实现循环**：先抽象批准提案为实施计划，然后逐原子子需求实现，组装补丁，运行确定性验证。失败反馈为精确修复指令。
    3. **质量评分**：八维加权得分（属性幻觉22%、正确性循环迭代18%、人工干预18%、DSL检查修正8%等），确保自动化可靠性。

#### 开发智能体（离线模型轨道）
- **结构**：策略声明 → (代码实现 ↔ 验证) (最多3次重写) + 专家代理投票 → 执行（Python状态机，无LLM） → 最终审查（含因果链归因）。
- **可证伪归因**：要求每个策略声明因果机制和预期可观察量。最终审查单独判断每个因果链环节的状态（已验证/断裂/不明确）。只有所有环节已验证且AUC提升归因清晰才记录成功。
- **鲁棒执行**：对训练平台故障分类处理（确定性错误立即放弃，瞬时故障重试一次，LLM网关故障切换）。

#### 评估智能体
- **安全部署**：根据业务域、世界、桶范围正确映射实验流量，进行预实验平衡检查，采用灰度发布。
- **A/B判断与护栏否决**：
    - 使用方差缩减方法（CUPED、双重差分等）。
    - **护栏设计三原则**：
        1. 业务范围定制（不同业务域有自己的核心指标和阈值）。
        2. 复合经济交换指标（综合多个业务目标的LTV得分），避免单指标噪音触发否决。
        3. 阈值作为注意力信号而非绝对阻断（严重恶化硬阻断，中度信号升级处理）。
- **负结果资产化**：将失败的实验写为结构化记录（根因、阶段、业务目标、策略杠杆），供未来避免重蹈覆辙。

#### 驾驭进化层（SGPO）
- **SGPO-I（基于对话轨迹）**：
    1. 采样轨迹，提取评估指标。
    2. 评估代理撰写自然语言损失报告（语义梯度）。
    3. 精炼代理将梯度转换为局部驾驭编辑（只改目标子智能体的指令、验证规则、输出合同或工具使用规范）。
    4. 配对回放：用新旧驾驭分别在相同任务上运行，只有当改进分数ΔJ > ε且安全检查通过时才接受更新。
- **SGPO-II（基于代码回放）**：证据源从对话轨迹变为历史合请求（MRs）。过滤后得到人类批准的代码变更，将每个MR转换为只保留需求的任务，评估五维（语义正确性40%、需求覆盖25%、文件覆盖20%、安全默认10%、代码风格5%），只有加权≥4且语义正确性≥4才通过。失败时的语义梯度用于更新开发智能体的执行规则、评估教训、模式记忆和确定性预检查。
- **模型研究探索的自我进化**：三阶段探索（论文复现→模块消融→跨论文组合），通过记忆机制（论文级和模块级剪枝）和饱和检测（新颖性签名重复率>80%时结束）自动收缩搜索空间。结论需通过阈值门（至少两个独立运行）和对抗审查门才注入未来轮次。

    - 并发实验数为工程师的8倍，累计业务价值为3.7倍。
    - SGPO进化使系统自我增强：每次试验的轨迹反馈回智能体框架和基础模型，长期运行中系统越来越强。
4. **模型研究探索可行**：AgentX能自主复现论文、消融模块、跨论文组合，在工业数据上保持有效。
5. **干预机制重要性**：绝大多数失败原因在基础设施层面（参数资源冲突64.7%），而非智能体推理质量，提示优化方向。

## 七、优点

### 方法论亮点
- **完全闭环设计**：从想法到线上反馈再到系统进化，形成自驱动的飞轮。
- **多知识源融合**：综合利用实验历史、系统架构、数据分析和外部研究，避免了单纯依赖LLM生成。
- **可证伪归因**：模型开发轨道要求显式声明因果机制和可观察量，防止虚假归因。
- **护栏设计**：复合经济指标、业务域定制、阈值作为注意力信号等设计，在安全性和误报率之间取得良好平衡。
- **SGPO进化层**：通过配对回放保证安全性，迭代局部驾驭更新，不改变全局框架。

### 实验设计优点
- **生产级验证**：在真实快手推荐系统上部署，获取真实用户反馈，远超离线模拟。
- **多维度评估**：不仅报告AUC，还报告实验吞吐量、并发数、累计业务价值、平台和基础设施瓶颈分析。
- **消融分析**：深入分析了想法被拒绝的原因（表6）、编码失败原因（表7）、SGPO进化模式（图8、9）。

## 八、不足与局限

### 实验覆盖方面
- **模型研究实验仅使用AUC指标**：未覆盖更多指标（如NDCG、Recall、多样性、用户满意度等）。
- **生产环境未公开详细的统计显著性细节**：例如置信区间、p值、最小效应阈值等。
- **工业场景实验**仅针对广告投放和用户增长场景，未推广到其他业务线。
- **SGPO进化实验**使用了有限的轨迹和回放任务，长期进化效果尚未充分验证。

### 偏差风险
- **LLM依赖**：系统高度依赖LLM（如DeepSeek、Qwen等），其偏差和幻觉可能传播。
- **知识库更新滞后**：系统知识库依赖异源码抽取，若代码变更未及时反映，可能导致生成错误。
- **评估代理的评判偏差**：LLM作为judge可能误判，SGPO依赖配对回放防止退化，但无法完全消除。

### 应用限制
- **平台依赖**：AgentX的组件（DSL检查器、特征模式查询）是定制化的，迁移到其他推荐系统需大量适配。
- **计算成本**：运行三个智能体推理、干运行、A/B实验需要大量计算资源，未公开成本。
- **负结果资产化仍有限**：虽然记录了失败原因，但未验证这些资产在后续轮次中有效避免重复失败的比例。
- **人类审查仍是瓶颈**：尽管自动化程度高，但想法验证和某些决策仍需人工介入。

### 其他局限
- **未探讨多智能体间冲突与协调**：多个worker并行运行时，资源冲突（参数冲突占64.7%）是主要拒绝原因，系统应对此类冲突的机制不够深入。
- **自进化速度**：实际周进化（并发15->60，通过率15%->45%）虽显著，但是否可长期持续有待观察。
- **安全性**：尽管有护栏，但自动化的代码修改和实验发布仍可能引入风险。
